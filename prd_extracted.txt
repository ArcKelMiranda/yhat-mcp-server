YHAT MCP SERVER
Product Requirements Document
Acceso seguro y auditable a SQL Server para asistentes de IA

1. Resumen Ejecutivo
Yhat MCP Server es un Model Context Protocol (MCP) server propio que permite al equipo de análisis de datos de Yhat interactuar con Microsoft SQL Server a través de asistentes de IA (Claude, Cursor, VS Code) sin exponer más superficie de la estrictamente necesaria.
A diferencia de los MCPs genéricos de terceros, este servidor aplica capas explícitas de validación, autorización y auditoría, de modo que la IA únicamente puede leer (y, en casos controlados, escribir) sobre las tablas y procedimientos que el equipo autorice de forma expresa.
1.1 Problema
Los MCPs de terceros exponen demasiado esquema o carecen de control granular por tabla/rol.
El equipo necesita explorar datos con IA sin riesgo de modificaciones accidentales o maliciosas.
No existe trazabilidad de qué consultas ejecuta la IA ni quién las solicitó.
Varios clientes de Yhat operan en el sector financiero (NinetyOne, AMCS, EDR, Skandia), por lo que cualquier acceso asistido por IA a sus datos debe ser defendible ante una auditoría.
1.2 Solución
Un MCP server propio, ligero y extensible que:
Expone solo las tablas/vistas autorizadas mediante whitelist explícita.
Bloquea operaciones destructivas por defecto (modo solo-lectura).
Registra cada interacción para auditoría, con logs legibles y trazables por usuario.
Permite perfiles de acceso diferenciados (reader / writer / admin) por miembro del equipo.
No depende de servicios cloud obligatorios, minimizando costo y superficie de ataque adicional.
1.3 Alcance y No-Objetivos
Para evitar ambigüedad sobre qué cubre este proyecto, se listan explícitamente los no-objetivos de la v1:
No es un reemplazo de las herramientas de BI existentes (Power BI, Looker) — es un canal de exploración ad hoc para el equipo técnico.
No gestiona la creación/alteración de esquema (DDL) en ningún rol, incluido admin, en la v1.
No provee interfaz gráfica propia; el punto de entrada es siempre un cliente MCP (Claude Desktop, Cursor, VS Code).
No sustituye los controles de acceso ya existentes a nivel de SQL Server/AWS; los complementa.

6.2 Flujo de una Consulta
Recepción: el cliente MCP envía una petición (ej.: "dame las ventas del Q2").
Traducción: el LLM genera el SQL (o el usuario lo proporciona directamente).
Validación: el Query Guard analiza el SQL — rol autorizado, reglas de seguridad, tabla en whitelist.
Ejecución: si pasa validación, se ejecuta contra SQL Server con parámetros ligados, timeout y límite de filas.
Logging: se registra la consulta (metadatos, no el resultado completo), el usuario, el resultado y el tiempo.
Respuesta: se devuelve el resultado al LLM en formato legible.

9. Casos de Uso
UC-01: Exploración de datos por un analista junior
Actor: Analista con rol "reader".
Flujo: Pide a Claude "muéstrame las ventas del último mes por región". El MCP valida que la tabla reporting.ventas_mensuales está en la whitelist, ejecuta el SELECT, limita a 1000 filas, loggea la acción y devuelve los resultados.
Resultado: el analista obtiene datos sin riesgo de modificar nada.
UC-02: Carga de datos en staging
Actor: Analista senior con rol "writer".
Flujo: Pide "inserta los registros del CSV en staging.clientes_nuevos". El MCP detecta que es un INSERT, verifica que la tabla está permitida para writers, pide confirmación explícita, ejecuta y loggea.
Resultado: operación controlada y trazada.
UC-03: Intento de operación destructiva
Actor: cualquier usuario.
Flujo: la IA genera accidentalmente un DROP TABLE reporting.ventas. El Query Guard detecta la palabra clave DROP, bloquea la ejecución, devuelve un error claro al LLM y loggea un WARN.
Resultado: la base de datos está protegida; el usuario recibe una explicación.
UC-04: Auditoría semanal
Actor: Tech Lead.
Flujo: revisa logs/yhat-mcp-2026-07-09.log y ve que una analista ejecutó 45 queries esta semana, todas de lectura, con tiempo promedio de 0.8s; detecta que una query fue bloqueada por intentar un DELETE sin WHERE.
Resultado: visibilidad completa del uso de la herramienta.
UC-05: Intento de inyección SQL vía prompt (nuevo)
Actor: usuario (intencional o accidental) o LLM con un prompt manipulado.
Flujo: el texto generado intenta cerrar la consulta original e inyectar SQL adicional. Como la ejecución usa parámetros ligados (FR-18), la inyección se trata como dato literal y no como código; adicionalmente el Query Guard bloquea patrones sospechosos y loggea WARN.
Resultado: la inyección no altera la consulta ejecutada.

13. Criterios de Aceptación
El proyecto se considera listo para producción interna cuando:
Un analista nuevo puede clonar el repo, instalar dependencias y conectar su cliente MCP en < 15 minutos.
Una consulta SELECT sobre una tabla whitelist se ejecuta y devuelve resultados en < 2s (P95).
Una consulta DROP TABLE es bloqueada, loggeada y el usuario recibe un mensaje claro de error.
Un usuario con rol "reader" no puede ejecutar un INSERT aunque lo solicite explícitamente.
Los logs contienen al menos: timestamp, usuario, query completa, filas afectadas, éxito/error — sin exponer datos de resultado sensibles.
Ninguna credencial de base de datos existe en texto plano en el repositorio ni en los archivos de configuración versionados.
El servidor corre estable durante 7 días sin reinicios forzados en el entorno de prueba.
14. Anexo: Ejemplo de Configuración YAML
Ejemplo ilustrativo de yhat-mcp-config.yaml — las credenciales se resuelven por variable de entorno, nunca se escriben aquí:
database:
  host: ${YHAT_DB_HOST}
  name: ${YHAT_DB_NAME}
  encrypt: true
  password_env: YHAT_DB_PASSWORD

whitelist:
  - schema: reporting
    tables: [ventas_mensuales, clientes]
    mode: read_only
  - schema: staging
    tables: [clientes_nuevos]
    mode: read_write

roles:
  reader:  { schemas: [reporting] }
  writer:  { schemas: [reporting, staging] }
  admin:   { schemas: ["*"], require_confirmation: true }

limits:
  max_rows: 1000
  query_timeout_seconds: 30
  rate_limit_per_minute: 30

Control | Cómo se cumple
Gestión de secretos | Credenciales nunca en el YAML ni en el repositorio; se leen de variables de entorno o de un gestor de secretos (p. ej. AWS Secrets Manager, ya usado en la infraestructura de Yhat).
Cifrado en tránsito | Conexión a SQL Server con TLS habilitado (Encrypt=yes, TrustServerCertificate según entorno).
Mínimo privilegio | La cuenta de servicio que usa el MCP tiene únicamente permisos GRANT SELECT sobre la whitelist, y GRANT INSERT/UPDATE/DELETE solo sobre el esquema staging.
Prevención de inyección SQL | Toda ejecución usa parámetros ligados (parameterized queries); el Query Guard es una segunda capa, no la única defensa.
Minimización de datos en logs | Los logs registran metadatos de la query (usuario, tabla, filas afectadas, duración), no el contenido de los resultados.
Separación de entornos | Whitelists y roles distintos por entorno (dev/staging/prod); por defecto el MCP apunta a una réplica de solo lectura cuando exista.

Término | Definición
MCP | Model Context Protocol — protocolo estándar para conectar LLMs con fuentes de datos y herramientas.
Query Guard | Componente de validación de SQL que aplica reglas de seguridad antes de ejecutar.
Whitelist | Lista explícita de elementos permitidos (tablas, esquemas, stored procedures).
RBAC | Role-Based Access Control — control de acceso basado en roles de usuario.
Parameterized query / parámetros ligados | Técnica que separa el SQL de los datos de entrada, evitando inyección SQL.
DAB (Data API Builder) | Herramienta de Microsoft para exponer APIs sobre bases de datos; evaluada y descartada como alternativa (ver sección 1.4).
pyodbc | Driver ODBC para Python, estándar para conectar con SQL Server.
